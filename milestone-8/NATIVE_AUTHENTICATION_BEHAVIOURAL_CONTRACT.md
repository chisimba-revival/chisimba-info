# Milestone 8: Native authentication behavioural contract

## Status

Design scaffold only. The current LiveUser authentication path remains authoritative.

## Purpose

The native replacement must preserve externally observable Chisimba behaviour while
removing runtime dependence on the obsolete LiveUser implementation. Internal design
may change; user-visible and security-relevant behaviour may not change accidentally.

## Required login behaviour

- Accept the same canonical username forms as the current system.
- Reject unknown users and wrong passwords without revealing which condition occurred.
- Reject disabled, deleted, expired, or otherwise inactive accounts consistently.
- Preserve the existing post-login destination and module routing behaviour.
- Regenerate the session identifier before marking a session authenticated.
- Populate every session value still required by core modules.
- Record successful and failed login events without storing plaintext passwords.
- Preserve current account-locking behaviour until an intentional replacement is approved.

## Required authorisation behaviour

- Resolve direct and group-derived permissions using the current precedence rules.
- Preserve administrator and special-group semantics.
- Return the same result for anonymous, authenticated, inactive, and deleted users.
- Avoid permission caching that can outlive a logout or group-membership change.

## Password migration behaviour

- Verify currently deployed password hashes before any migration is enabled.
- Upgrade a legacy hash only after successful verification of the supplied password.
- Use `password_hash()` for replacement hashes.
- Never rewrite a hash following a failed login.
- Make migration observable in audit logs but invisible to the user.
- Provide a tested rollback strategy that does not strand upgraded accounts.

## Session behaviour

- Preserve inactivity timeout, absolute timeout, logout, and browser-close behaviour.
- Remove all authenticated state during logout.
- Prevent session fixation by regenerating the identifier at authentication boundaries.
- Maintain compatibility with modules that read legacy session keys during transition.

## Error behaviour

- User-facing failures must remain generic.
- Detailed causes belong only in protected logs.
- Repository or database failures must fail closed.
- Authentication failures must not emit PHP warnings, notices, stack traces, or SQL.

## Non-goals for the scaffold phase

- No login-controller changes.
- No service-container or object-factory registration.
- No database writes beyond files generated by the scaffold script itself.
- No feature flag activation.
- No removal or modification of LiveUser.
